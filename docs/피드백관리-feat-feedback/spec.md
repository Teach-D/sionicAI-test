# 기능 명세서 — 사용자 피드백(Feedback) 관리

- **브랜치**: `feat/feedback`
- **작성일**: 2026-07-14

---

## 1. 개요

사용자가 특정 대화(`Chat` — 개별 Q&A 쌍)에 대한 긍정/부정 피드백을 남기는 기능을 제공한다.
피드백은 대화 품질 개선을 위한 데이터로 활용되며, 관리자가 처리 상태를 관리할 수 있다.

---

## 2. 도메인 모델

엔티티 필드·DB 타입·인덱스는 [erd.md](./erd.md)를 단일 출처로 한다.

### 피드백(Feedback) 생성 규칙

- 일반 사용자(`MEMBER`)는 **자신이 생성한 스레드에 속한 Chat**에만 피드백을 생성할 수 있다.
- 관리자(`ADMIN`)는 **모든 대화**에 피드백을 생성할 수 있다.
- 동일 사용자 + 동일 대화 조합에는 피드백이 **1개만** 생성 가능하다 (중복 방지).
- 하나의 대화에는 서로 다른 사용자(예: 소유자 + 관리자)가 각각 피드백을 남길 수 있다.

### 피드백 상태(Status)

| 상태 | 설명 |
|---|---|
| `PENDING` | 피드백 생성 직후 초기 상태 |
| `RESOLVED` | 관리자가 처리 완료로 변경한 상태 |

- 상태 변경은 `ADMIN`만 가능하다.
- 상태 변경 방향: `PENDING → RESOLVED` 단방향만 허용. 역방향 전환은 Out-of-scope.

## 4. 구현 범위

### In-scope

- `Feedback` 엔티티 및 `FeedbackRepository`
- 피드백 생성 (`POST /api/feedbacks`)
- 피드백 목록 조회 (`GET /api/feedbacks`) — 정렬, 필터링, 페이지네이션
- 피드백 상태 변경 (`PATCH /api/feedbacks/{feedbackId}/status`) — `ADMIN` 전용

### Out-of-scope

- 피드백 삭제
- 피드백 내용(긍정/부정) 수정
- 피드백 단건 조회 (목록 조회로 대체)
- 알림 발송 (상태 변경 시 이메일/푸시 등)
